### 3. Fix aplicat

El problema era que el runtime real depèn que systemd executi:

```ini
ExecStart=/opt/redismanager/bin/redismanager-ctl launch %i
User=%i
Group=%i
```

És a dir: `launch` corre com l'usuari cPanel del compte, no com `root`.

Per tant, RedisManager no pot assumir un model `root-only` absolut. El fix actual deixa això explícit:

- només `launch` pot córrer sense `root`
- les comandes administratives continuen sent `root-only`
- el camí no-root només llegeix l'estat
- els permisos instal·lats tornen a ser compatibles amb aquest model

### 4. Normalització de `.user.ini`

També es va detectar que versions anteriors del plugin havien escrit el bloc de session locking a `.user.ini` sense marcadors, mentre que la versió nova usa un bloc marcat.

L'estat actual del codi:

- `deploy_session_locking()` normalitza el fitxer abans d'escriure
- `remove_session_locking()` també neteja el format legacy
- s'evita l'acumulació de blocs duplicats

## Coses que un altre agent no hauria d'assumir malament

- No assumir que `redismanager-ctl` és completament `root-only`.
- No tancar permisos de runtime sense revisar abans la unitat systemd real.
- No assumir que `.user.ini` sempre està en format nou amb marcadors.
- No assumir que cal rerun l'instal·lador complet per desplegar un fix petit en producció.
- No matar processos en producció “en bloc” si abans no s'ha distingit el procés gestionat per systemd dels orfes.
